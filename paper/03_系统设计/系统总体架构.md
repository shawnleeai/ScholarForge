# 3.2 系统总体架构

基于3.1节的需求分析，本节设计ScholarForge系统的总体架构。系统采用微服务架构，结合现代化的技术栈，实现高可用、高扩展的智能科研论文写作平台。

## 3.2.1 架构设计原则

### （1）微服务架构选择

**选择理由**：

**业务复杂性**：ScholarForge涉及多种业务领域（文献检索、AI写作、协同编辑、用户管理等），微服务架构可以将复杂系统拆分为独立的服务，降低开发和维护难度。

**团队规模**：采用微服务架构，不同团队可以独立开发和部署各自的服务，提高开发效率。

**技术异构性**：不同服务可以采用最适合的技术栈。例如，AI服务使用Python，前端服务使用Node.js，充分利用各语言的优势。

**独立扩展**：不同服务的负载特性不同，微服务架构允许针对热点服务单独扩容，优化资源使用。

**独立部署**：服务的独立部署能力支持快速迭代，新功能可以快速上线而不影响其他服务。

### （2）前后端分离设计

**分离模式**：
- **前端**：React 18单页应用，负责UI渲染和用户交互
- **后端**：FastAPI微服务集群，负责业务逻辑和数据处理
- **通信**：RESTful API + WebSocket实时通信

**优势**：
- 前后端可以独立开发、测试、部署
- 前端可以灵活适配多种终端（Web、移动端）
- 后端API可以服务多种客户端

### （3）云原生设计理念

**容器化**：所有服务使用Docker容器化打包，确保环境一致性。

**编排部署**：使用Kubernetes进行容器编排，实现自动扩缩容和故障恢复。

** DevOps**：建立CI/CD流水线，实现自动化测试和部署。

**可观测性**：集成日志、监控、链路追踪，确保系统可观测。

## 3.2.2 微服务架构

### （1）服务划分

根据功能需求，将系统划分为以下微服务：

**网关服务（Gateway）**：
- 功能：统一入口、路由转发、认证鉴权、限流熔断
- 端口：8000
- 技术：Nginx / Kong / 自研FastAPI Gateway

**用户服务（User Service）**：
- 功能：用户注册、登录、信息管理、权限控制
- 端口：8001
- 技术：FastAPI + PostgreSQL

**论文服务（Paper Service）**：
- 功能：论文CRUD、版本管理、大纲管理
- 端口：8002
- 技术：FastAPI + PostgreSQL

**文献服务（Article Service）**：
- 功能：文献检索、文献管理、PDF解析
- 端口：8003
- 技术：FastAPI + Elasticsearch + PostgreSQL

**AI服务（AI Service）**：
- 功能：AI写作、文献综述、研究问答、文本润色
- 端口：8004
- 技术：FastAPI + LangChain + 多LLM Provider

**协作服务（Collaboration Service）**：
- 功能：实时协作编辑、评论、版本历史
- 端口：8005
- 技术：WebSocket + Yjs + Redis

**推荐服务（Recommendation Service）**：
- 功能：文献推荐、期刊推荐、每日推送
- 端口：8006
- 技术：FastAPI + Redis + 机器学习模型

**查重服务（Plagiarism Service）**：
- 功能：相似度检测、查重报告
- 端口：8007
- 技术：FastAPI + SimHash/MinHash

**通知服务（Notification Service）**：
- 功能：消息推送、邮件通知、站内信
- 端口：8008
- 技术：FastAPI + RabbitMQ + 第三方推送

**Agent服务（Agent Service）**：
- 功能：多Agent调度、Agent间通信
- 端口：8009
- 技术：FastAPI + 自定义Agent框架

### （2）服务间通信

**同步通信（RESTful API）**：
适用于实时性要求高的场景，如用户登录、数据查询。

调用方式：
```python
# 服务A调用服务B
async def call_service_b(params):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://service-b:8002/api/endpoint",
            json=params
        )
        return response.json()
```

**异步通信（消息队列）**：
适用于耗时操作和解耦场景，如AI生成、邮件发送。

消息队列选型：
- **Redis Stream**：轻量级，适合简单场景
- **RabbitMQ**：功能丰富，支持复杂路由
- **Apache Kafka**：高吞吐量，适合大数据量

使用示例：
```python
# 发布消息
async def publish_task(task_data):
    await redis.xadd("ai_tasks", {"data": json.dumps(task_data)})

# 消费消息
async def consume_tasks():
    while True:
        messages = await redis.xread({"ai_tasks": "$"}, block=0)
        for message in messages:
            await process_task(message)
```

### （3）API网关设计

**核心功能**：

**路由转发**：
根据请求路径将请求路由到对应的服务。

```
/api/v1/users/*    → User Service (8001)
/api/v1/papers/*   → Paper Service (8002)
/api/v1/articles/* → Article Service (8003)
/api/v1/ai/*       → AI Service (8004)
...
```

**认证鉴权**：
- JWT Token验证
- 权限检查
- 用户上下文传递

**限流熔断**：
- 限流：防止单个用户/服务占用过多资源
- 熔断：服务故障时快速失败，防止级联故障

**负载均衡**：
- 轮询（Round Robin）
- 加权轮询
- 最少连接

## 3.2.3 技术栈选型

### （1）前端技术栈

**框架**：React 18
- 选择理由：组件化、生态丰富、性能优秀

**语言**：TypeScript
- 选择理由：类型安全、IDE支持好、维护性强

**UI组件库**：Ant Design 5
- 选择理由：企业级组件、文档完善、社区活跃

**状态管理**：Zustand + React Query
- Zustand：轻量级状态管理
- React Query：服务端状态管理，缓存、同步

**构建工具**：Vite
- 选择理由：启动快、热更新快、配置简单

**实时协作**：Yjs
- 选择理由：成熟的CRDT库，支持多种编辑器

### （2）后端技术栈

**框架**：FastAPI（Python 3.11）
- 选择理由：高性能、自动生成文档、类型提示支持

**ORM**：SQLAlchemy 2.0
- 选择理由：功能强大、支持异步、社区活跃

**验证**：Pydantic
- 选择理由：数据验证、序列化、与FastAPI集成好

**AI框架**：LangChain
- 选择理由：LLM应用开发的标准框架，链式调用

**LLM Provider**：
- OpenAI GPT-4/GPT-3.5
- Anthropic Claude
- 国产模型（DeepSeek、Moonshot、Stepfun）

### （3）数据库与存储

**关系数据库**：PostgreSQL 16
- 用途：存储用户、论文、文献等结构化数据
- 特性：支持JSON、全文检索、窗口函数

**缓存数据库**：Redis 7
- 用途：会话缓存、热点数据缓存、消息队列
- 特性：高性能、支持多种数据结构

**图数据库**：Neo4j 5
- 用途：知识图谱、文献关系网络
- 特性：Cypher查询语言、图算法支持

**搜索引擎**：Elasticsearch 8
- 用途：文献全文检索、日志分析
- 特性：分布式、实时搜索、聚合分析

**对象存储**：MinIO
- 用途：PDF、图片、附件存储
- 特性：S3兼容、高可用

### （4）基础设施

**容器化**：Docker
**编排**：Kubernetes
**CI/CD**：GitHub Actions / GitLab CI
**监控**：Prometheus + Grafana
**日志**：ELK Stack（Elasticsearch + Logstash + Kibana）
**链路追踪**：Jaeger

## 3.2.4 数据流设计

### （1）用户请求处理流程

```
用户请求 → API网关 → 认证鉴权 → 路由转发 → 业务服务 → 数据库 → 返回结果
                ↓
         限流/熔断检查
```

详细流程：
1. 用户发送请求到API网关
2. 网关进行JWT Token验证
3. 网关检查限流规则
4. 网关根据路径路由到对应服务
5. 业务服务处理请求
6. 如需AI处理，调用AI服务
7. 如需数据，查询数据库
8. 返回结果给用户

### （2）AI写作处理流程

```
用户请求写作 → AI服务 → 提示工程 → 调用LLM → 流式返回 → 用户界面
                  ↓
              RAG检索（如需要）
                  ↓
              知识库/文献库
```

详细流程：
1. 用户在编辑器中请求AI写作
2. 前端通过WebSocket发送请求到AI服务
3. AI服务构建提示词（Prompt Engineering）
4. 如需背景知识，进行RAG检索
5. 调用LLM API（流式调用）
6. 将生成的内容流式返回给前端
7. 前端实时显示生成内容

### （3）实时协作编辑流程

```
用户A编辑 → Yjs文档 → WebSocket → 协作服务 → WebSocket → 用户B/C/D...
                ↓
            持久化存储（IndexedDB/数据库）
```

详细流程：
1. 用户A在编辑器中输入内容
2. Yjs捕获编辑操作，生成更新
3. 更新通过WebSocket发送到协作服务
4. 协作服务将更新广播给所有协作者
5. 用户B/C/D的Yjs客户端应用更新
6. 同时更新持久化存储

### （4）多Agent协同流程

```
用户任务 → Agent服务 → 任务分解 → 任务分配 → 各Agent执行 → 结果整合 → 返回用户
                ↓
            调度器（调度算法）
                ↓
            冲突检测/解决
```

详细流程：
1. 用户提交复杂任务（如"生成文献综述"）
2. Agent服务接收任务
3. 协调Agent将任务分解为子任务
- 子任务1：检索相关文献（检索Agent）
- 子任务2：阅读并提取要点（阅读Agent）
- 子任务3：生成综述文本（写作Agent）
- 子任务4：审校内容（审校Agent）
4. 调度器根据Agent能力和负载分配任务
5. 各Agent并行/串行执行任务
6. 协调Agent整合各Agent的结果
7. 最终结果返回给用户

## 3.2.5 部署架构

### （1）开发环境

**本地开发**：
- 前端：npm run dev（Vite热更新）
- 后端：各服务独立运行（uvicorn）
- 数据库：Docker Compose启动本地数据库

**工具**：
- VS Code + 远程开发插件
- Postman / Swagger UI（API测试）
- pgAdmin / Redis Desktop Manager（数据库管理）

### （2）测试环境

**部署方式**：Docker Compose

**服务组成**：
- 各业务服务容器
- Nginx反向代理
- PostgreSQL容器
- Redis容器
- MinIO容器

### （3）生产环境

**部署方式**：Kubernetes集群

**架构**：
```
[负载均衡器]
     ↓
[Nginx Ingress]
     ↓
[Service Mesh（Istio）]
     ↓
[业务Pods]
     ↓
[持久化存储]
```

**高可用设计**：
- 多副本部署（每个服务至少2个副本）
- 数据库主从复制 + 读写分离
- Redis哨兵模式/集群模式
- 跨可用区部署

**本节小结**：

本节设计了ScholarForge系统的总体架构。系统采用微服务架构，划分为10个核心服务，通过API网关统一管理。技术栈选型遵循现代化、成熟稳定的原则，前端采用React 18 + TypeScript，后端采用FastAPI + Python 3.11，数据库采用PostgreSQL + Redis + Neo4j的组合。数据流设计涵盖了用户请求、AI写作、协作编辑、多Agent协同等核心场景。部署架构支持开发、测试、生产多环境，生产环境采用Kubernetes实现高可用部署。
